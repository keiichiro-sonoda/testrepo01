# クロスワード解答ガイド（写真→解答→正解検証）

このドキュメントは、**「問題画像を Claude に渡せば、解いて正解検証まで行う」**ための手順書です。
**カタカナ・10×10・二重枠A〜Jボーナス** といったフォーマット（毎回ほぼ同じ）を想定しています。

> **このリポジトリのツールは汎用、データは問題ごと。**

> - 問題ごと（手元で作る／**公開リポジトリには含めない**）：黒マス配置・各カギの答え・二重枠の位置・完成図画像

## 結論：手順は汎用。ただし「目視確認」を要所で挟むのが前提

完全自動ではありません。**自動検出 → 機械的な検算 → 怪しい所だけ目視で補正**、という
半自動フローが、誤りを出さないために重要です。下の手順をその順で実行してください。

## 手順（新しい問題画像を受け取ったら）

### 0. 画像を切り出す
盤面とカギ文章を、それぞれ高解像度でクロップ・拡大して読む（`PIL`）。
斜め・影・折り目があるとOCRが落ちるので、必要なら部分ごとに拡大する。

### 1. 黒マスを検出する（`detect_grid.py`）
```bash
python3 detect_grid.py photo.jpg --search X0 Y0 X1 Y1 --n 10 --overlay ov.png
```
- セル中央領域の **輝度の中央値（median）** で黒/白を判定する（手書きや汚れに強い）。
- 出力の `median 輝度` 表で、**120〜180 付近の境界値セルは必ず目視確認**する。
- `ov.png`（検出を写真に重ねた画像）を必ず見て、黒マスの過不足を直す。

### 2. 番号付けがカギ一覧と一致するか検算する（最重要チェック）
```python
from grid import Board
b = Board(black=BLACK, n=10)
ok, diff = b.verify_numbering(expected_across=[...], expected_down=[...])
```
- `expected_*` はカギ一覧（ヨコ／タテ）に載っている番号の集合。
- **一致しなければ黒マスが間違っている。** `diff` の missing/extra を見て黒マスを修正 → 再検算。
- ここが合えば、盤面構造（黒マス・番号・どのマスがどの語に属するか）は確定。

### 3. カギを全部読み取る
番号・文章・マス数（エントリ長）を、ヨコ／タテすべて書き出す。

### 4. カギを解いて盤面に置く（`Board.place`）
```python
b.place(2, "A", "____")   # 長さ不一致・交差矛盾は例外で弾かれる
```
- 知識で解く。**ただし知識だけで確定しない。交差で必ず裏を取る。**
- 矛盾が出たら答えを見直す。曖昧なカギ（固有名詞・時事）は交差から逆算する。

### 5. 全マス充足を確認
```python
assert b.missing() == []   # 未充足の白マスが無い
```

### 6. ボーナス（二重枠A〜J）を抽出して検算
- 二重枠セルの位置を、**解答を写真に重ねたオーバーレイ**で1つずつ照合する。
- A→J の順に文字を読む。**読んで意味が通らなければ、枠位置か文字が間違っている。**

### 7. 完成図を作図（`render_solution.py`、手元用）
解答を記した spec.json から、黒マス・番号・解答・二重枠を matplotlib で描く。
※ この spec.json と出力画像は解答そのものなので、公開リポジトリには置かない。

## 難しかったポイント＝今後の落とし穴

> ここが本ガイドの核心。同じ罠は次回も踏みやすい。例示はすべて**架空**にしてある。

### ① 盤面の幾何（黒マス・交差）の確定が最難関
目視だと斜め・手書きでズレる。**必ず検出ツール＋番号一致チェックで固める。**

### ② 薄くスキャンされた黒マスを取りこぼす
黒マスでも、スキャンが薄いと中央値輝度が ~150 になり、自動検出（閾値120）が白と誤判定する。
→ すると番号付けがカギ一覧と合わなくなって**検出ミスが発覚**し、目視で黒に修正できる。
**境界値セルは必ず目視。番号一致チェックが安全網。**

### ③ 小書き仮名は「大書き1マス」（最重要の言語ルール）
新聞クロスワードは ッ ャ ュ ョ ー を **大書きで1マス**に入れる。
- 例（架空）：「ジャッジ」→ **ジ・ヤ・ツ・ジ** の4マス、「ショート」→ **シ・ヨ・ー・ト** の4マス
- これを忘れると交差が解けない／**偽の矛盾**が出る。
- 典型：ヨコの語とタテの語が交差マスで衝突して見えるが、**小書きを大書きに揃えると両立**する。
  「ッ を ツ と置く」に気づけるかが鍵。

### ④ 手書きの答えは無視する
写真に既に書き込まれた答えは**間違いの可能性がある**ので信用しない。

### ⑤ 各文字は「2つの交差」で二重確認できる
ヨコとタテが交わるので、1マスは2エントリで決まる。
**知識ベースの初手が外れても、交差が正しい仮名を強制して訂正してくれる。**
固有名詞・時事で記憶が曖昧でも、カギの文面＋交差から逆算して確定できることが多い。
（→ 詳しくは「誤り訂正符号的な見方」の節）

### ⑥ ボーナスは「枠の位置特定」と「意味の検算」が要
- 二重枠のタグ(A〜J)はマスの角にあり、どのマスを囲むか紛らわしい。
  → **解答オーバーレイで照合**して確定。
- A→J を読んで一見無意味でも、**区切ると意味のある語句**になることがある
  （カタカナを単語の切れ目で区切ると、漢字熟語などが現れる）。**最後は必ず意味が通るかで検算。**
- ボーナスは問題イントロ（時事の前振り等）とテーマが一致することが多く、検算のヒントになる。

### ⑦ 最終検証は4点セット
1. 全交差が矛盾ゼロ（`place` が一度も例外を出さない）
2. 全白マス充足（`missing()` が空）
3. 番号付けがカギ一覧と一致（`verify_numbering`）
4. ボーナスが意味を成す

この4つが揃って初めて「解けた」と判断する。

## 誤り訂正符号的な見方（なぜ多少のノイズに耐えるか）

クロスワードは**冗長性をもった誤り訂正符号**に近い。各白マスは「ヨコの語・タテの語・カギの意味」で
多重に拘束され、高確度マスから制約が伝播して未確定マスが埋まる（LDPC符号の反復復号に近い）。

ただし**効く誤りと効かない誤りが質的に違う**：

| ノイズの種類 | 符号理論の対応 | 耐性 |
|---|---|---|
| 文字の読み違い（ある1マスの仮名の誤認） | 置換誤り | **強い**。交差が矛盾を出して局所訂正 |
| 黒マスの誤判定（構造の読み違い） | **同期ずれ**（挿入・欠失） | **弱い**。語長がずれ番号付け全体が崩壊 |

- 置換型（読み違い）は交差＋意味拘束で復号できる。記憶が空白の時事問題でも交差で当たることがある。
- 構造型（黒マス誤り）は局所では直らず、**大域チェック（`verify_numbering`）でしか検出できない**。
- 冗長度は不均一：交差が多くカギも分かるマスは鉄板、無交差マスや両カギとも難問のマスは脆い。
- 拘束自体が壊れる（白黒が読めない／カギ文が読めない領域）と容量超過で破綻＝「ノイズだらけは無理」。

## 画像品質と試行錯誤について（現実）

**完全自動化はできない。** 写真・スキャンの品質はまちまちで、解像度・傾き・影・折り目・
スキャンのかすれ・手書きの写り込みによって `画像→黒マス` と `画像→カギ文章` の精度が変わる。
この2ステップだけは、**毎回その場でクロップ位置・拡大率・閾値を合わせ込む試行錯誤**が要る。
（盤面が確定した後の番号付け・交差チェック・作図は決定的で揺るがない）

### ただし試行錯誤は「収束」する
**`verify_numbering`（番号一致チェック）が客観的なゴール判定**になる：
黒マスを直して番号がカギ一覧と一致するまで繰り返せばよい。「合うまで直す」基準があるので、
画質が悪くても最終的に正解にたどり着ける。

### 画質に応じて変える手
- bbox 自動推定が外れる → `--bbox` で手動指定、`--search` で探索範囲を絞る
- 黒判定がかすれ黒を落とす → `--black-thr` を上げる／境界セルを個別に拡大して目視
- 文字が小さい・つぶれる → 盤面・カギを部分ごとに高倍率でクロップして読み直す
- 斜め・歪み → 必要なら台形補正してから検出

### 人間側でできる品質改善（精度が上がる撮り方）
1. **平らなスキャン > 斜め写真**（歪みが最大の敵）
2. **高解像度・明るく・影なし**
3. **盤面とカギを別々に接写**して複数枚渡す（小さい文字対策）
4. できれば**手書きを書き込む前**の状態（誤記が混ざらない）

完璧な画像は不要。上記が揃うほど試行錯誤の回数が減って速くなる、という関係。
